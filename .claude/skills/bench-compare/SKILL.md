# /bench-compare

Purpose: when a change claims a performance impact (principle C6.7 â€” "performance-sensitive changes require benchmark comparisons"), produce the before/after numbers in one command, without making the user manually stash, checkout, run, un-stash, and re-run.

Usage:
- `/bench-compare BenchmarkDictionary` â€” compares current working tree vs. `merge-base HEAD upstream/master` (falls back to `origin/master` if upstream missing).
- `/bench-compare BenchmarkDictionary <baseline-ref>` â€” compare against an explicit ref (commit, tag, branch).
- `/bench-compare BenchmarkDictionary --args "-wi 1 -i 2 -f 1 -r 5s -w 5s"` â€” pass extra JMH args. **Always use short warmup/iteration flags for a first pass**; defaults run for hours or days.

**Time expectations.** Pinot benchmarks are not quick. Default JMH config in `pinot-perf` is 8 warmup Ă— 60s + 8 measurement Ă— 60s Ă— 5 forks per parameter combination â€” a single benchmark method's `@Benchmark` can report an ETA of multiple days. The skill will refuse to run without either: (a) explicit `--args` that reduce warmup/iteration counts, or (b) the user confirming they really do want the full default run.

## Procedure

1. **Locate the benchmark.** Glob for `pinot-perf/**/<benchmarkName>.java`. If zero or multiple matches, report and stop. The benchmark class must be under `pinot-perf`.

2. **Resolve the baseline ref.**
   - Default: `git merge-base HEAD upstream/master`. If the `upstream` remote isn't defined, fall back to `origin/master`. If neither resolves, ask the user for an explicit ref.
   - If the user passed a ref, validate it with `git rev-parse --verify <ref>`.

3. **Prepare output directory.** `mkdir -p .bench-compare/` and append it to the repo's `.gitignore` if not already there. Produce two files: `baseline-<short-sha>.txt` and `current-<short-sha-or-WIP>.txt`.

4. **Warn and confirm.** Benchmarks take real time. Inspect `--args` â€” if the user hasn't passed iteration controls, warn that the default suite can take hours to days and suggest a starter like `-wi 1 -i 2 -f 1 -r 5s -w 5s`. Print an estimate of the pair of runs (rough: a 5s-warmup Ă— 5s-measurement Ă— 1 fork run takes ~30â€“120s per `@Benchmark` method after the Pinot-side `@Setup` completes; `@Setup` alone can run for 1â€“10 minutes for benchmarks that build segments). Ask the user to confirm.

5. **Build pinot-perf in a baseline worktree.** This avoids touching the working tree:
   ```
   git worktree add /tmp/pinot-bench-baseline <baseline-ref>
   (cd /tmp/pinot-bench-baseline && ./mvnw -pl pinot-perf -am package -DskipTests)
   ```
   The package goal produces both the jar and the `pinot-perf/target/pinot-perf-pkg/bin/pinot-<BenchmarkClass>.sh` launcher (generated by `appassembler-maven-plugin`). Use the launcher â€” it wires up the right classpath, JVM flags, and heap sizing.

6. **Run the baseline benchmark.**
   ```
   /tmp/pinot-bench-baseline/pinot-perf/target/pinot-perf-pkg/bin/pinot-<BenchmarkClass>.sh <extra-args>
   ```
   If the script doesn't exist, fall back to direct Java invocation:
   ```
   java -Xms4G -Xmx4G -cp '/tmp/pinot-bench-baseline/pinot-perf/target/pinot-perf-pkg/lib/*' org.apache.pinot.perf.<BenchmarkClass> <extra-args>
   ```
   For the vector suite (`BenchmarkVectorIndex`) use the `exec:java` form from `pinot-perf/README.md`; the appassembler script isn't generated for it.

   Save stdout+stderr to `.bench-compare/baseline-<short-sha>.txt`.

7. **Build and run the current tree.** Same steps against the main repo (no worktree switching needed â€” `./mvnw` and the `pinot-perf-pkg` both live in the current working directory):
   ```
   ./mvnw -pl pinot-perf -am package -DskipTests
   pinot-perf/target/pinot-perf-pkg/bin/pinot-<BenchmarkClass>.sh <extra-args>
   ```
   Save to `.bench-compare/current-<sha-or-WIP>.txt`.

8. **Clean up the worktree.** `git worktree remove /tmp/pinot-bench-baseline --force`. Do this even if steps 6 or 7 failed.

9. **Diff the results.** Parse JMH's table output (the `Benchmark ... Score Error Units` lines) from both files. Produce a table:
   ```
   Benchmark          Baseline (ops/s)    Current (ops/s)    Î”        Î”%
   foo.methodA        1234.5 Â± 12.1       1478.2 Â± 15.3      +243.7   +19.7%
   foo.methodB         987.6 Â±  8.0        992.1 Â±  7.2       +4.5     +0.4%
   ```
   Use "ops/s", "ns/op", or whatever unit JMH emits â€” don't convert.

10. **Report.** Print the table. Flag any benchmark where `|Î”%| > 2Ă—error%` as likely a real change (otherwise probably noise). Include the paths to the raw files so the user can share them in a PR.

## Notes

- Worktrees require a clean `.git`. If the repo is in the middle of a rebase/merge, abort with a clear message.
- The generated `pinot-<BenchmarkClass>.sh` scripts hard-code `-Xms24G -Xmx24G`. That's fine on a workstation but will OOM a small machine. If the user is on a laptop with <32GB RAM, suggest the direct `java -cp` fallback with `-Xms4G -Xmx4G`.
- JMH's `-l` (list benchmarks) flag **does not help here** â€” Pinot benchmark classes have custom `main()` entry points that wrap `Runner.run(...)` and ignore it. There is no way to do a "fast sanity check" of a benchmark invocation short of actually running one iteration.
- `exec:java` in `pinot-perf` spawns a separate JVM; that's fine â€” JMH does its own JVM forking anyway.
- Benchmarks must run on the same hardware, same JDK, same OS load. Warn the user if they're on battery power or running other heavy processes.
- Do not `sleep` between runs. JMH's own warmup iterations handle JIT warmup; adding sleep changes nothing meaningful.
- If the benchmark's output format isn't plain JMH (e.g. `BenchmarkVectorIndex` writes a custom report), don't try to parse it â€” just save both outputs and tell the user where they are, with a note that manual comparison is needed.
- Never use `git stash` instead of a worktree. Stash can be lost if the second build fails and the user doesn't know to pop it.
